# AGENTS.md

이 폴더는 영어 단어 학습용 Obsidian 문서 저장소다. 소프트웨어 프로젝트가 아니라, 어근별 단어 분류와 테마별 어휘 학습을 위한 WIKI 스타일 자료를 만들고 관리하는 공간이다.

## 역할

- 사용자가 제시한 영어 단어를 어근, 접두사, 접미사 등으로 분석한다.
- 분석 결과를 먼저 사용자에게 간단히 알려준다.
- 사용자가 특정 어근을 더 공부하고 싶어 하면, 같은 어원을 공유하는 단어들을 찾아 변화 과정과 의미 연결을 정리한다.
- 사용자가 특정 주제나 상황을 제시하면, 그 테마에서 학습 가치가 큰 난이도 있는 단어들을 골라 정리한다.
- 최종 학습 문서는 Obsidian에서 읽기 좋은 Markdown 문서로 작성한다.

## 저장 위치

- 어근별 결과물은 반드시 `Roots/` 서브폴더에 저장한다.
- 새 문서 파일명은 위키 스타일에 맞게 대표 어근만 사용한다.
- 예: `Roots/leg.md`
- 사용자가 처음 제시한 기준 단어는 파일명에 넣지 말고 문서 안의 "출발 단어" 또는 첫 항목으로 기록한다.
- 어근 문서가 새로 생기면 `Roots/_Index.md`에 링크를 추가한다.
- 관리용 문서는 앞에 `_`를 붙여 어근 문서와 구분한다. 예: `Roots/_Index.md`
- 실제 `index` 어근 문서가 필요하면 `Roots/index.md`처럼 일반 어근 문서로 별도 작성한다.
- 테마별 결과물은 반드시 `Themes/` 서브폴더에 저장한다.
- 테마 문서 파일명은 주제를 짧고 읽기 쉬운 영어 kebab-case로 만든다. 예: `Themes/law-and-rights.md`, `Themes/emotion-and-attitude.md`
- 테마 문서가 새로 생기면 `Themes/_Index.md`에 토픽 타이틀과 문서 링크를 추가한다.
- `Themes/_Index.md`에는 각 테마의 단어를 모두 나열하지 않는다. 토픽 타이틀, 링크, 짧은 설명만 둔다.

## 단어 선정 범위

- 같은 어근 단어를 찾을 때는 TOEFL, TOEIC, IELTS, SAT, ACT 수준에서 자주 보거나 학습 가치가 큰 단어를 중심으로 고른다.
- 너무 희귀한 고전어, 전문 학술어, 법률/의학/신학 등 특정 분야의 깊은 전문어는 사용자가 요청하지 않는 한 제외한다.
- 어원적으로 관련이 있더라도 현대 시험 어휘 학습에 도움이 적으면 간단히 언급하거나 생략한다.
- 한 문서에 너무 많은 단어를 넣기보다 핵심 단어와 파생 관계가 분명한 단어를 우선한다.

## 테마 문서 단어 선정 범위

- 사용자가 특정 주제를 주면, 그 주제와 관련된 난이도 있는 학습 단어를 선별한다.
- 기준은 TOEFL, TOEIC, IELTS, SAT, ACT 수준에서 학생들이 제법 어려워하고 공부할 가치가 큰 단어다.
- 너무 쉬운 기본 단어는 제외한다. 예: `good`, `bad`, `make`, `go`처럼 주제 설명에는 쓰이지만 별도 암기 가치가 낮은 단어.
- 너무 지엽적인 전문어, 희귀어, 특정 분야 내부에서만 쓰이는 단어는 사용자가 요청하지 않는 한 제외한다.
- 기본 분량은 최대 20개 정도다. 중요한 단어가 적으면 20개를 억지로 채우지 않는다.
- 중요한 단어가 20개를 넘고 주제 학습상 가치가 분명하면 20개를 넘겨도 된다.
- 정부, 경제, 과학처럼 범위가 큰 테마는 하위 묶음으로 나누어 25-35개 정도까지 확장해도 된다.
- 큰 테마에서 특정 하위 영역이 비어 보이면 단어 수 제한보다 학습 구조의 균형을 우선한다.
- 같은 어원에서 나온 변형형은 의미가 크게 다르지 않으면 따로 적지 않는다. 예: `analyze`, `analysis`, `analytical` 중 테마 학습에 가장 필요한 형태 중심으로 정리.
- 다른 테마에 이미 등장한 단어라도 이 테마에서 학습 가치가 있으면 중복해서 적어도 된다.
- 테마 문서에서는 문서 간 링크를 굳이 만들지 않는다. 링크보다 해당 테마에서의 의미, 사용 맥락, 주의점 기록을 우선한다.

## 기본 작업 흐름

1. 사용자가 예를 들어 `legal`이라는 단어를 제시하면, 먼저 `legal = leg + al`처럼 구성 요소를 확인해 알려준다.
2. 사용자가 `leg` 어원을 더 공부하고 싶어 하면, `legal`, `loyal`, `privilege`처럼 같은 계열로 설명할 수 있는 단어들을 모은다.
3. 각 단어의 의미 변화, 형태 변화, 현대 영어에서의 쓰임을 간단히 정리한다.
4. 정리한 내용을 `Roots/leg.md` 같은 파일로 저장한다.
5. `Roots/_Index.md`에 `[[leg]]` 링크와 짧은 설명을 추가한다.

## 테마 문서 작업 흐름

1. 사용자가 주제를 제시하면, 먼저 해당 테마의 범위를 짧게 정한다.
2. 그 테마에서 시험·교양 어휘로 학습 가치가 큰 단어를 선별한다.
3. 기본적으로 20개 이내로 정리하되, 핵심 단어가 적으면 적은 개수로 끝낸다.
4. 각 단어는 해당 테마에서 중요한 의미와 사용 맥락을 중심으로 설명한다.
5. 일반 의미와 특정 맥락의 의미가 다르면 반드시 구분한다. 예: 일반 의미로는 `charge` = 요금/책임/돌진이지만, 법률 맥락에서는 `혐의`, `기소 내용`으로 쓰임.
6. 헷갈리는 단어, register 차이, 시험에서 자주 묻는 의미 차이, 분야별 특수 의미는 "주의할 점" 또는 단어 항목 안에 명확히 기록한다.
7. 정리한 내용을 `Themes/토픽.md` 같은 파일로 저장한다.
8. 새 테마 문서를 만들면 `Themes/_Index.md`에 토픽 타이틀과 `[[토픽]]` 링크, 짧은 설명을 추가한다.

## 자동 문서화 기준

- 사용자가 단어 존재 여부를 묻거나 새 단어를 제시했을 때, 해당 어근 문서가 없고 TOEFL, TOEIC, IELTS, SAT, ACT 수준에서 충분히 학습 가치가 있으면 새 어근 문서를 자동으로 만든다.
- 자동 생성 대상은 같은 어근에서 시험·교양 어휘를 4개 이상 자연스럽게 묶을 수 있거나, 기준 단어 하나만으로도 파생어·비교어 학습 가치가 큰 경우다.
- 문서화 가치가 낮거나 어원이 불확실하면 바로 만들지 말고, 후보와 보류 이유를 짧게 알린다.
- 새 문서를 자동으로 만들 때도 먼저 기준 단어의 어근 분석을 간단히 제시하고, 생성 후 `Roots/_Index.md`와 `Roots/_Lexicon.json`을 갱신한다.

## 속도 최적화 작업 흐름

- 새 단어가 들어오면 먼저 `Roots/_Index.md`에서 대표 어근 문서 존재 여부만 확인한다.
- 이미 있는 어근이면 해당 문서 1개만 열어 수정한다.
- 새 어근이면 `_Index.md`에 없는 것만 확인하고 바로 `Roots/대표어근.md`를 만든다.
- 관련 문서 비교는 충돌 가능성이 크거나 의미 구분이 필요한 경우에만 1-2개로 제한한다.
- 사용자가 "다른 후보", "전체적으로", "중복 확인", "재그룹화"처럼 넓은 탐색을 요청하지 않는 한 `Roots/` 전체를 훑지 않는다.
- 단어가 어느 문서에 이미 들어 있는지 확인할 때는 가능하면 `scripts/find_word.py`를 먼저 사용한다. 이 스크립트의 작은 출력만 보고 필요한 문서만 연다.
- `Roots/_Lexicon.json`은 단어 항목 역색인이다. 문서를 만들거나 항목을 추가한 뒤에는 가능하면 `python3 scripts/build_lexicon.py`로 갱신한다.
- `scripts/find_word.py`와 `scripts/doc_stats.py`는 기본적으로 최신 `Roots/_Lexicon.json`을 우선 사용한다. 색인이 없거나 오래되었을 때만 Markdown 전체를 훑는다.
- 본문 언급까지 넓게 찾을 필요가 있을 때만 `scripts/find_word.py --scan ...`을 사용한다.
- 새 어근 문서를 만든 뒤 `_Index.md`에 항목을 넣을 때는 가능하면 `python3 scripts/add_index_entry.py root '설명'`을 사용한다. 설명에 백틱이 들어가므로 셸에서 작은따옴표로 감싼다. `_Index.md`의 주변 문구가 자주 바뀌므로 수동 패치 문맥 불일치를 줄이기 위함이다.
- 새 문서 생성과 `_Index.md` 수정은 가능하면 별도 단계로 처리한다. 한 패치에 묶으면 인덱스 문맥 불일치 때문에 새 문서 생성까지 롤백될 수 있다.
- Obsidian CLI가 사용 가능하면 보조 검색과 화면 확인에 활용한다. 예: `obsidian search query=diagnosis path=English-Word/Roots limit=5`, `obsidian outline path=English-Word/Roots/dia.md total`, `obsidian read path=English-Word/Roots/dia.md`.
- Obsidian CLI는 앱과 통신해야 하므로 Codex 샌드박스 안에서는 실패할 수 있다. 실패하면 샌드박스 밖 실행 또는 `scripts/open_in_obsidian.py`를 사용한다.
- 문서 편집 자체는 여전히 로컬 Markdown 파일을 직접 수정하는 방식을 기본으로 한다. CLI의 `append`, `create`, `delete`, `move`는 사용자가 명시하거나 Obsidian 상태 반영이 꼭 필요할 때만 쓴다.
- 작업 후 수정한 노트를 Obsidian에서 열어볼 때는 `python3 scripts/open_in_obsidian.py Roots/대표어근.md`를 사용할 수 있다.

## 여러 컴퓨터 작업 원칙

- 사용자는 여러 Mac에서 번갈아 작업할 수 있다. 스크립트, 배포 방식, 운영 규칙을 바꾸면 해당 변경을 GitHub 배포 저장소에도 반드시 반영한다.
- 작업 지침이 바뀌면 iCloud 원본의 `AGENTS.md`에 남기고, 다른 Mac 사용자가 따라야 하는 실행 절차는 Workspace 저장소의 `README.md`에 남긴다.
- `scripts/` 안의 도구를 고치거나 새로 만들면 예시 명령, dry-run 여부, 주의할 점을 `README.md`나 스크립트 help에 같이 기록한다.
- 테마 heading을 어근 문서와 연결할 때는 먼저 `python3 scripts/link_theme_roots.py --list-ambiguous`로 dry-run을 보고, `python3 scripts/link_theme_roots.py --write --list-ambiguous`로 적용한다.
- 복수 어근 후보가 나오는 항목은 표제어 뒤에 후보 링크를 나열한다. 예: `aqueduct ([[aqua#aqueduct|aqua]], [[duc#aqueduct|duc]])`.
- 특정 테마에서 후보 링크가 너무 복잡하거나 학습 흐름상 하나만 필요하면 사람이 수동으로 줄인다.
- 배포 전에는 Workspace 저장소에서 `git pull --ff-only`가 선행되어야 한다. `publish/publish_notes.sh`도 이 단계를 수행한다.
- 작업이 끝나면 가능하면 `publish/publish_notes.sh "설명적인 커밋 메시지"`로 GitHub 저장소와 GitHub Pages까지 갱신한다.

## 문서 스타일

- 한국어 설명을 기본으로 한다.
- 영어 단어, 어근, 접사는 인라인 코드로 표시한다.
- Obsidian에서 연결하기 좋도록 관련 문서가 생기면 위키 링크를 사용할 수 있다.
- 어근 문서 상단에는 출발 단어, 핵심 의미, 관련 어근 또는 헷갈리는 어근 링크를 간단히 둔다.
- 내용은 암기용으로 너무 장황하지 않게, 하지만 어원적 연결은 분명하게 쓴다.
- 문체는 짧은 단정형을 선호한다. 불필요한 서술 어미를 줄이고, `가리킨다`보다 `가리킴`, `~라는 뜻이 된다`보다 `~의 뜻`처럼 압축해서 쓴다.
- 단어별 항목에는 가능하면 다음을 포함한다.
  - 단어
  - 구성 또는 역사적 형태
  - 핵심 의미
  - 어원적 연결
  - 간단한 예문 또는 사용 맥락

## 테마 문서 스타일

- 한국어 설명을 기본으로 한다.
- 영어 단어는 인라인 코드로 표시한다.
- 테마 문서는 어원 설명보다 "이 주제에서 왜 중요한 단어인지"를 우선한다.
- 문체는 짧은 단정형을 선호한다.
- 테마 문서 상단에는 주제 범위만 간단히 둔다.
- 선정 기준, 작업 기준, 핵심 주의 같은 관리용 문구는 개별 테마 문서에 쓰지 않는다.
- 단어별 항목에는 가능하면 다음을 포함한다.
  - 단어
  - 해당 테마에서의 핵심 의미
  - 사용 맥락 또는 짧은 예문
  - 주의할 의미 차이 또는 특수 용법
- 역사, 정치, 사회운동, 과학사처럼 배경지식이 중요한 테마는 필요하면 "역사적 고유명사" 또는 "주요 고유명사" 섹션을 따로 둔다.
- 고유명사는 인물·사건·판례·법·운동·시대명을 중심으로 고르고, 단순 암기보다 독해에서 자주 배경지식으로 작동하는 것만 넣는다.
- 문서 간 링크는 필수가 아니다. 테마 학습 흐름을 방해하지 않는 경우에만 제한적으로 사용한다.

## 문서 재그룹화

- 한 어근 문서의 단어 항목이 대략 20-30개 이상으로 늘어나면 재그룹화를 검토한다.
- 단순히 개수만 보지 않고, 의미 이미지가 둘 이상으로 뚜렷하게 갈라지는지도 함께 본다.
- 재그룹화할 때는 기존 어근 문서를 허브 문서로 남기고, 하위 문서를 `어근-갈래.md` 형식으로 만든다.
- 예: `tract.md`를 허브로 두고 `tract-pull.md`, `tract-handle.md`, `tract-trace.md`처럼 분리.
- 하위 문서를 만들면 허브 문서와 `Roots/_Index.md`에 링크를 추가한다.
- 검색용 형태 문서는 짧게 유지하고 핵심 허브로 연결한다. 예: `sess.md` → `[[sed]]`.

## 어원 분리 원칙

- 철자나 현대 의미가 비슷해도 어원이 다르면 같은 어근 문서에 본문 항목으로 섞지 않는다.
- 어원이 다른 단어는 별도 대표 어근 문서로 분리한다. 예: `servare` 계열은 `serv.md`, `servire/servus` 계열은 `servire.md`.
- 한 단어가 여러 어원 갈래를 가진 동형이의어이면, 문서 안에서 갈래를 명확히 나누거나 필요하면 별도 문서로 분리한다.
- 어원이 다른데 학습상 같이 언급할 필요가 있으면 본문 단어 목록이 아니라 "비교 단어" 또는 "주의할 형태"에서만 짧게 설명한다.
- 새 문서를 만들기 전에 비슷한 철자 문서가 있더라도, 실제 역사적 어원이 같은지 확인한 뒤 병합 여부를 결정한다.

## 주의

- 어원이 불확실하거나 논쟁적인 경우 단정하지 않고, "가능성이 크다", "전통적으로 연결된다", "현대 형태에서는 직접 분해하기 어렵다"처럼 조심스럽게 표시한다.
- 현대 영어 단어를 무리하게 현재 형태 기준으로만 쪼개지 않는다. 필요한 경우 라틴어, 프랑스어, 고대 영어 등 중간 형태를 함께 설명한다.
- 사용자가 이후 추가 지침을 주면 이 파일에 반영한다.
